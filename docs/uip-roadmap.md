# uIP-uya 开发路线图（按当前实现状态更新）

本文档用于把“使用 uya 重写 C 版 uIP 协议栈”的目标，整理为一份与当前代码一致的开发路线图。

路线图只服务于一个目标：

> **继续补齐 C 版 uIP 已有功能，不扩展，不重构成新协议栈。**

---

## 1. 当前阶段判断

当前仓库已经不再只是文档或纯骨架状态。

根据现有代码与测试，当前已完成：

- 核心类型层
- 编译期配置层
- 单实例全局状态
- ARP 最小闭环
- IPv4/ICMP 最小闭环
- UDP 最小收发闭环
- TCP 最小握手、收发、关闭、重传与定时路径
- host-side 回归测试基线

因此，后续工作重点不应再写成“创建基础模块”，而应写成：

- 对照 C 版补齐缺失功能
- 收敛临时测试辅助接口
- 保持文档与代码一致

---

## 2. 开发原则

### 2.1 只补齐 C 版已有功能

后续所有开发必须满足：

- 如果 C 版 uIP 没有，就不新增
- 如果只是为了“更现代”“更漂亮”，就不做
- 如果只是为了测试方便，不能改变核心模型

### 2.2 当前目录结构保持不变

当前实际模块布局为：

```text
src/uip/
  uip_arch/
    mod.uya
  uip_arp/
    mod.uya
  uip_conf/
    mod.uya
  uip_core/
    mod.uya
  uip_types/
    mod.uya
  uip_types_test.uya
```

后续路线图以这个结构为准，而不是旧文档中的单文件布局。

### 2.3 先补主流程，再补外围

优先顺序：

1. `uip_core/mod.uya`
2. `uip_arp/mod.uya`
3. `uip_types/mod.uya`
4. `uip_conf/mod.uya`
5. `uip_arch/mod.uya`
6. 测试补齐

原因是：`uip_core/mod.uya` 最接近 C 版 `uip.c`，也是当前功能差距最可能存在的地方。

---

## 3. 当前已完成里程碑

## M1：类型、配置、状态基础

已完成内容：

- `uip_types/mod.uya`
- `uip_conf/mod.uya`
- `uip_arch/mod.uya` 的基础能力
- `uip_core/mod.uya` 中 `UipState` 与 `uip_init()`

当前结论：已具备表达 uIP 核心状态与报文头的能力。

## M2：ARP 最小闭环

已完成内容：

- ARP 表
- cache 更新
- aging
- ARP request/reply 最小处理
- `uip_arp_out()` 的 hit/miss 路径

当前结论：已具备 ARP 外接协作最小闭环。

## M3：IPv4 / ICMP / UDP 最小闭环

已完成内容：

- 最小 IP 入站处理
- ICMP echo reply
- UDP 输入匹配
- UDP 周期发送
- UDP checksum 生成/校验

当前结论：已具备最小 IPv4/ICMP/UDP 闭环。

## M4：TCP 最小闭环

已完成内容：

- listen table 管理
- passive open 最小握手
- active open 最小握手
- established 下最小收发
- close / fin / ack / rst 基本路径
- periodic 下最小重传、超时、timewait 路径

当前结论：已具备最小 TCP 子集。

---

## 4. 后续里程碑

## M5：补齐 `uip.c` 当前未覆盖分支

### 目标

继续对照 C 版 `uip.c`，补齐当前尚未覆盖的主流程分支与边界情况。

### 范围

仅包括 C 版已有内容，例如：

- 当前 TCP 状态机中未覆盖的分支
- 输入报文的边界检查细节
- 与现有测试未覆盖对应的原版处理路径

### 不包括

- 新协议
- 新 API 设计
- 新抽象层

### 验收标准

- 文档能明确列出新增覆盖的是哪一段 C 版能力
- 测试能覆盖对应路径
- 不引入超出原版的新语义

---

## M6：补齐当前缺失的原版配置与 helper

### 目标

仅在核心流程确有需要时，补齐原版配置项和 helper。

### 可能范围

- 原版已有但当前未落地的配置常量
- 更接近原版语义的 checksum / add32 helper
- `uiplib` 风格辅助函数（仅在 C 版已有时）

### 验收标准

- 新增内容必须能对应到 C 版具体模块
- 不以“方便未来扩展”为理由提前加入无关能力

---

## M7：收敛测试辅助接口

### 目标

逐步减少当前为 host-side 测试引入的过渡接口对核心设计说明的干扰。

### 范围

当前代码中已明确属于过渡性质的内容，如：

- `uip_send_at()`
- `uip_udp_bind()`
- 某些 direct materialization helper

### 原则

- 如果这些接口继续保留，文档必须明确写成“测试辅助”
- 如果原版已有更接近的语义，应尽量收敛到原版风格

### 验收标准

- 文档中不把测试辅助接口写成正式对外 API
- 核心行为模型仍然围绕 `uip_process(flag)` 与全局状态

---

## 5. 当前不做的事情

以下内容当前明确不做：

- 不新增 socket API
- 不新增异步框架式网络抽象
- 不新增动态连接池
- 不新增多 buffer 体系
- 不新增原版没有的协议特性
- 不新增“更现代”的网络分层设计
- 不先写示例应用来反向驱动架构扩张

---

## 6. 文档维护要求

后续每次实现更新后，文档必须同步更新以下内容：

1. 当前目录结构
2. 当前已实现功能列表
3. 当前未实现功能列表
4. 哪些接口是 transitional/test helper
5. 哪些能力已经有测试覆盖

文档不允许：

- 提前声明未实现能力
- 把路线图写成“未来可能扩展”清单
- 把测试辅助接口包装成正式架构设计

---

## 7. 当前最重要的结论

当前项目最准确的阶段描述是：

> **uya 版已经完成了 C 版 uIP 的一个最小可运行子集，并建立了测试基线；后续工作应严格围绕补齐 C 版既有功能展开，而不是继续扩展。**
