# uIP-uya

一个使用 **uya** 从零重写 **C 语言版 uIP 协议栈** 的实验性项目。

本项目聚焦于三个核心方向：

- **忠实重写 uIP**：以 C 版 uIP 的真实实现结构为基准进行语义级重写
- **零 GC 路线**：保持资源可预测、静态分配、固定容量的数据模型
- **可测试演进**：在不背离原始 uIP 架构的前提下，为实现过程建立可验证的测试基线

---

## 项目目标

uIP-uya 旨在探索：

1. 如何使用 uya 语言重写 C 版本的 uIP，而不是重新设计一套新的协议栈架构。
2. 如何在保留 uIP 轻量、紧凑、静态资源模型的前提下完成语义迁移。
3. 如何在单主缓冲区、固定连接表、编译期裁剪这些约束下，实现与原版接近的行为与资源占用。
4. 如何让这样一个偏底层、偏紧耦合的协议栈实现仍然具备可测试性。

这个项目既是一次协议栈重写实践，也是一次对 **uya 在受限系统编程场景中的表达能力** 的验证。

---

## 为什么选择 uIP

uIP 是一个经典的轻量级 TCP/IP 协议栈，最初就是为小型 8 位 / 16 位单片机设计的。

它的重要价值在于：

- 实现足够小，适合作为从零重写对象
- 设计取舍非常鲜明，适合研究资源受限场景下的协议实现方法
- 代码中清晰体现了“以 RAM/ROM 为先”的工程哲学

对本项目来说，uIP 不只是“一个协议栈功能集合”，更是一种非常具体的实现方法：

- 单一主缓冲区
- 单核主处理流程
- TCP / UDP / IP 紧耦合实现
- 固定容量连接表
- 编译期配置裁剪
- ARP 外接处理模型

本项目希望保留的正是这些关键特征。

## 设计原则

### 1. 忠实重写，而不是重新架构

本项目的目标不是借 uIP 之名设计一个“更现代”的协议栈，而是尽量保留 C 版 uIP 的核心设计，例如：

- `uip_buf + uip_len` 主缓冲区模型
- `uip_process(flag)` 单核入口
- 固定容量 TCP / UDP / ARP 状态表
- 驱动与协议栈直接协作的调用方式
- ARP 在核心之外独立处理

因此，代码组织可以适配 uya，但架构思想应尽量接近原版。

### 2. 零 GC 与静态资源模型

uIP 的价值之一就在于资源占用极低，而这一点必须依赖：

- 静态分配
- 固定容量数组
- 单主缓冲区
- 极少中间对象
- 避免动态内存与运行时扩容

uya 版也将尽量沿用这一资源模型，而不是引入更重的抽象层。

### 3. 可测试演进

虽然 uIP 原始实现偏紧耦合、偏单体，但重写过程仍然需要测试约束，至少应覆盖：

- 报文输入输出行为
- 连接状态迁移
- 校验和计算
- 定时器与重传路径
- ARP / IP / ICMP / UDP / TCP 的最小闭环

重点不是把协议栈改造成“为测试而设计”的新架构，而是在忠实实现的基础上建立测试能力。

## 当前代码状态

当前仓库处于早期实现阶段，已完成的内容主要包括：

- `uip_types.uya`：核心协议类型、头部结构、状态枚举
- `uip_conf.uya`：第一版编译期配置常量
- `uip.uya`：全局状态、初始化逻辑、连接/发送相关骨架
- `uip_arp.uya`：ARP 表、ARP 输入处理、ARP 老化、最小出站路径

尚未完成的核心部分包括：

- `uip_process(flag)` 的完整主处理流程
- IPv4 / ICMP / UDP / TCP 主路径联通
- TCP 状态机、重传与定时逻辑
- 更完整的测试基线与驱动调用示例

也就是说，当前目标仍然是**先把骨架搭对，再逐步补齐协议行为**。

## 仓库结构

```text
src/
  uip/
    uip.uya
    uip_arch.uya
    uip_arp.uya
    uip_conf.uya
    uip_types.uya

docs/
  README.md

  uip-porting-plan.md
```

主要模块含义：

- `src/uip/uip.uya`
  - 对应 C 版 `uip.c`
  - 承载核心全局状态与主入口骨架
- `src/uip/uip_types.uya`
  - 对应 `uip.h` 中的主要类型、头部与状态定义
- `src/uip/uip_conf.uya`
  - 对应 `uipopt.h` / `uip-conf.h` 风格的配置模型
- `src/uip/uip_arp.uya`
  - 对应 `uip_arp.c`
  - 实现 ARP 外接处理逻辑
- `src/uip/uip_arch.uya`
  - 对应 `uip_arch.h`
  - 用于承载字节序、校验和或平台相关能力

## 开发思路建议

建议按接近 C 版 uIP 的方式推进，而不是按现代分层协议栈的方式推进：

1. 先建立 `uip` 核心状态与主缓冲区模型
2. 重建 `uip_process(flag)` 等价主流程
3. 完成 ARP / IPv4 / ICMP 最小闭环
4. 再加入 UDP 支持
5. 最后推进 TCP 状态机、重传与定时处理

这样可以确保实现路线始终围绕“重写 uIP”这一目标，而不是逐步演变成另一套协议栈。
